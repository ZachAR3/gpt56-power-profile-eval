# 5.5 High Evaluation: Power-Profile Overhaul

## 1. Executive summary

Verdict: **Requires significant fixes**.

The overhaul has a good direction: it separates desired/applied/observed state, routes hardware calls through the backend service, persists editable profile definitions, improves GUI selection behavior, and attempts to address real Intel PPD/RAPL/EPP races. However, the current implementation is not ready to merge or release. Two power-profile defects are release-blocking:

- The service masks `power-profiles-daemon` at startup, but normal profile application still requires `powerprofilesctl set` to succeed. After initialization, profile changes and startup reconciliation can therefore report partial failure and skip the new delayed RAPL/EPP correction path.
- The direct MSR writer encodes PL2 incorrectly: PL2 enable and time-window bits are shifted out of the 64-bit value, so the write can leave PL2 disabled or malformed.

There are also medium-severity lifecycle and testability issues around fire-and-forget delayed rewrites, unreported rewrite failures, and new hardware methods that bypass the adapter's `root_path` test boundary. A separate pre-existing or environment-specific problem blocked application-level validation: the `HardwareCommandQueue` timed out even for `lambda: 42` under the available Python 3.14 environment, causing backend/application tests to hang or fail.

Minimum readiness bar: fix the PPD lifecycle/apply contract, correct and unit-test MSR encoding, make delayed rewrites ordered/cancellable/observable, and restore reliable backend tests.

## 3. Architecture and behavior overview

The power-profile system currently has these main parts:

- `ConfigStore` owns persisted desired state in `/var/lib/honor-control/state.toml`. `PowerState.profile` is the user's desired manual profile. `PowerState.profiles` stores built-in and custom profile definitions with PL1/PL2, governor, EPP, PPD profile, turbo, and max performance.
- `ApplicationService` owns use-case orchestration. Public D-Bus/CLI/GUI calls enter methods such as `set_power_profile`, `save_power_profile`, `configure_auto_switch`, and background auto-switch logic.
- `SnapshotStore` publishes immutable `PowerSnapshot` values containing desired profile, applied profile, observed hardware summary, AC state, auto-switch settings, and profile definitions.
- `HardwareCommandQueue` is intended to serialize all blocking hardware operations on one daemon worker thread.
- `HonorToolsAdapter` implements production hardware behavior by wrapping the `honor-tools` dependency and direct Linux interfaces.
- `FakeHardware` implements the same port for tests and session-bus development.
- `PowerPage` renders active profile selection, profile editing, auto-switch controls, and telemetry.

Profile application flow:

1. `set_power_profile(profile)` validates against `self._config.state.power.profiles`.
2. `_apply_power_profile(name, persist_desired=True)` checks `get_power_capability`.
3. It builds a definition dict from the persisted `PowerProfileState`.
4. It calls `HonorToolsAdapter.apply_power_profile` through the command queue.
5. The adapter builds a temporary `honor.config.Config`, injects the selected profile, and calls `honor.power.apply_profile`.
6. `honor.power.apply_profile` writes RAPL via sysfs, sets governors, calls `powerprofilesctl set`, writes EPP, writes final governor, and writes Intel turbo/max-performance controls.
7. The adapter aggregates returned per-mechanism booleans into `governor_ok`, `epp_ok`, `ppd_ok`, `rapl_ok`, and `misc_ok`.
8. Only if all five are true does `ApplicationService` update `_last_applied_power_profile`, optionally persist desired state, refresh the power snapshot, and schedule `_delayed_power_rewrite`.

New delayed correction flow:

1. `_apply_power_profile` schedules `_delayed_power_rewrite(profile)` with `asyncio.ensure_future`.
2. The task sleeps 0.5 seconds.
3. It calls `write_rapl_msr(pl1_uw, pl2_uw)` and `rewrite_epp(epp, governor)` through the command queue.
4. Return values are ignored; only raised exceptions are logged.

Startup flow:

1. `initialize()` loads config.
2. It refreshes all snapshots.
3. It calls `stop_competing_power_daemons`.
4. It calls `_reconcile_power_profile`, which reapplies persisted desired state if the current snapshot's applied profile differs.

Operating-system integration:

- PPD integration is still through `powerprofilesctl`.
- RAPL direct correction writes `/dev/cpu/0/msr`, MSRs `0x606` and `0x610`.
- EPP correction enumerates `/sys/devices/system/cpu/cpu[0-9]*/cpufreq/energy_performance_preference`.
- Governor correction writes `/sys/devices/system/cpu/cpu*/cpufreq/scaling_governor`.
- Startup daemon control runs `systemctl stop` and `systemctl mask` for `power-profiles-daemon` and `intel_lpmd`.
- HWP dynamic boost writes `/sys/devices/system/cpu/intel_pstate/hwp_dynamic_boost`.

State ownership:

- Desired profile and definitions live in `ConfigStore`.
- Last service-applied profile lives in `_last_applied_power_profile`.
- Hardware-observed profile is inferred from PPD's active profile in `HonorToolsAdapter.read_power`.
- The published applied profile is `_last_applied_power_profile or pw.applied_profile`, so once the service applies a profile it can mask later observed drift in snapshots.

Error and recovery paths:

- Initial `honor.power.apply_profile` partial failures become `OperationResult.partial`.
- Startup reconciliation logs failures but does not stop service startup.
- Delayed MSR/EPP rewrite failures are logged only if they raise; false return values are not surfaced.
- No rollback exists for partial power-profile application.

## 4. Findings

### PP-001: Startup masks PPD but profile application still requires `powerprofilesctl set`

- Severity: High
- Confidence: High
- Category: Functional regression, service lifecycle, OS integration
- Introduced by overhaul: Newly introduced
- Affected code:
  - `honor_control/backend/application.py:155-165`
  - `honor_control/backend/application.py:348-389`
  - `honor_control/backend/hardware.py:832-840`
  - `honor_control/backend/hardware.py:844-896`
  - Dependency path: `../honor-tools/honor/power.py:_set_ppd` and `apply_profile`

Description:

`initialize()` now stops and masks `power-profiles-daemon` before startup reconciliation and before any later user profile changes. But the profile apply path still treats `ppd_ok` as mandatory success. `ppd_ok` is true only when `honor.power.apply_profile` successfully runs `powerprofilesctl set`. With PPD stopped and masked, `powerprofilesctl set` is expected to fail or be unable to activate the daemon.

Evidence and reasoning:

- `initialize()` calls `self._hw.stop_competing_power_daemons` at `application.py:164`.
- `stop_competing_power_daemons` runs `systemctl stop power-profiles-daemon` and `systemctl mask power-profiles-daemon` at `hardware.py:874-883`.
- `HonorToolsAdapter.apply_power_profile` sets `"ppd_ok": raw.get("ppd_ok") is True` at `hardware.py:836`.
- `_apply_power_profile` requires `ppd_ok` along with governor/EPP/RAPL/misc at `application.py:353`.
- If any mechanism is false, it returns partial and does not persist the profile or schedule delayed rewrite (`application.py:382-389`).
- The overhaul plan explicitly says missing PPD is not failure when a profile has no PPD requirement, but current profiles always carry a `ppd_profile` and current application still requires PPD success.

Failure scenario:

On a supported Honor platform, the service starts, masks PPD, then a user selects "Performance". `honor.power.apply_profile` writes some sysfs values but `powerprofilesctl set performance` fails because PPD is masked. The D-Bus/GUI/CLI result is partial, desired profile is not persisted, and the new direct MSR/EPP correction is never scheduled. Auto-switch has the same failure mode and will repeatedly fail on AC/battery transitions.

Recommended fix:

Choose one coherent ownership model:

- If Honor Control replaces PPD, remove `powerprofilesctl set` from required application success after PPD is stopped/masked, and do not model PPD as a required mechanism.
- If PPD remains part of profile selection, do not mask it at startup. Instead apply PPD first, then perform direct corrections with an explicit race strategy.
- Persist and report whether PPD is intentionally disabled by Honor Control, and update capability/docs accordingly.

Suggested tests:

- Unit test with a hardware fake returning `ppd_ok=False` after `stop_competing_power_daemons` to prove profile application still succeeds when PPD is intentionally disabled, or that daemon masking is not performed before a PPD-required apply.
- Application test for startup reconciliation with desired `performance` and observed `balanced`.
- Auto-switch test where the selected profile is applied after initialization and delayed rewrite is scheduled.

### PP-002: Direct RAPL MSR writer shifts PL2 enable/time-window bits out of the register

- Severity: High
- Confidence: High
- Category: Functional bug, hardware interface
- Introduced by overhaul: Newly introduced
- Affected code:
  - `honor_control/backend/hardware.py:961-1025`
  - Specifically `hardware.py:1014-1019`

Description:

`write_rapl_msr` builds a 64-bit value for MSR `0x610`, but the high-half PL2 fields are encoded as if they were already in absolute register bit positions and are then shifted left by 32 again. As a result, `1 << 47` and `tw2 << 49` are shifted beyond bit 63 and removed by the final `& 0xFFFFFFFFFFFFFFFF` mask. PL2 units remain, but the PL2 enable bit and PL2 time-window bits are lost.

Evidence and reasoning:

Current code:

```python
hi = (
    pl2_units
    | (1 << 47)  # PL2 enabled
    | (tw2 << 49)
)
val = (lo | (hi << 32)) & 0xFFFFFFFFFFFFFFFF
```

The inline non-mutating calculation using the current formula produced:

- `pl1_enable True`
- `pl2_enable False`
- `tw2 0`
- `pl2_units 400`

This follows directly from the code's own layout: if `hi` is shifted by 32, its enable bit must be relative to the high half (`1 << 15`), not absolute bit 47.

Failure scenario:

A performance profile requests PL2 55 W. The delayed rewrite writes MSR `0x610` with PL2 power bits present but PL2 disabled/time-window zeroed. Depending on CPU behavior, burst limit may not apply as intended, may fall back to previous/default behavior, or may produce unstable power behavior. The service logs "RAPL MSR written" and returns success despite the malformed value.

Recommended fix:

Encode PL2 fields consistently. For example, either build absolute fields without shifting `hi`, or make `hi` a 32-bit half:

```python
hi = pl2_units | (1 << 15) | (tw2 << 17)
val = lo | (hi << 32)
```

Also mask `pl1_units` and `pl2_units` to their documented field widths before OR-ing.

Suggested tests:

- Pure unit test for `encode_pkg_power_limit(old, pl1_uw, pl2_uw, units)` that asserts PL1 enable, PL2 enable, PL1/PL2 power fields, and preserved time windows.
- Boundary tests for 3 W and 150 W.
- Test that invalid PL values do not open or write the MSR.

### PP-003: Delayed rewrites can apply stale profile data after a newer profile change

- Severity: Medium
- Confidence: High
- Category: Concurrency, state synchronization
- Introduced by overhaul: Newly introduced
- Affected code:
  - `honor_control/backend/application.py:183-184`
  - `honor_control/backend/application.py:367-421`

Description:

Each successful profile apply schedules `_delayed_power_rewrite(profile)` after 0.5 seconds. The previous delayed task is not cancelled before assigning `self._pending_power_task` to the new task, and the task does not check that its profile is still current before writing MSR/EPP. It runs outside `_mutation_lock`.

Evidence and reasoning:

- New tasks are assigned at `application.py:371-373`.
- Only the last task reference is retained.
- Shutdown cancels only the current `self._pending_power_task` at `application.py:183-184`.
- `_delayed_power_rewrite` takes a `PowerProfileState` object, sleeps, and then writes its PL/EPP values without checking the desired/applied profile at `application.py:406-419`.

Failure scenario:

User applies `silent`, then quickly applies `performance` within 0.5 seconds. The `silent` delayed task can wake after the `performance` apply and write silent PL1/PL2/EPP over the newer profile. The later `performance` delayed task may eventually correct it, but there is a visible window of wrong power state; if the service shuts down, the second task is cancelled, or the second rewrite fails, stale values can remain.

Recommended fix:

Use a generation token or profile name check. Cancel and await any prior delayed rewrite before scheduling a new one, or make `_delayed_power_rewrite` verify that the captured generation/profile still matches current applied intent immediately before each hardware write. Track all tasks or use a supervised background controller instead of a single overwritten task reference.

Suggested tests:

- Fake hardware test that applies two profiles rapidly and asserts only the latest profile's delayed PL/EPP values are written.
- Shutdown test proving no stale delayed rewrite runs after shutdown begins.
- Auto-switch race test for AC/battery transition followed by manual apply.

### PP-004: Delayed rewrite failures are not reflected in operation results or snapshots

- Severity: Medium
- Confidence: High
- Category: Error handling, observability, recovery
- Introduced by overhaul: Newly introduced
- Affected code:
  - `honor_control/backend/application.py:367-421`
  - `honor_control/backend/hardware.py:898-1025`

Description:

The new delayed correction is the mechanism that is supposed to make RAPL/EPP "stick", but `_apply_power_profile` returns success before it runs. `_delayed_power_rewrite` ignores boolean return values from both `write_rapl_msr` and `rewrite_epp`, and no snapshot refresh records whether the correction succeeded.

Evidence and reasoning:

- `_apply_power_profile` returns success immediately after scheduling the task (`application.py:371-381`).
- `_delayed_power_rewrite` awaits both calls but does not inspect returned `False` (`application.py:408-419`).
- `write_rapl_msr` returns `False` for missing `/dev/cpu/0/msr`, invalid PL values, permission errors, or other `OSError` (`hardware.py:976-986`, `hardware.py:1021-1025` plus later exception handling).
- `rewrite_epp` returns `False` if any CPU EPP path fails (`hardware.py:931-959`).

Failure scenario:

The system lacks the `msr` kernel module or `/dev/cpu/0/msr` permissions. The initial sysfs apply reports success, the user sees "Profile applied", but the delayed MSR correction fails and the service neither marks the power domain stale nor reports partial application. The system may drift back to PPD/kernel defaults while the UI continues to show the last service-applied profile due to `_last_applied_power_profile`.

Recommended fix:

Make the correction phase observable. Options:

- Include delayed correction in the operation before returning success, if latency is acceptable.
- Or publish a pending/apply-details state and update the snapshot when the delayed correction completes.
- Treat `False` from either delayed call as a warning/partial state and expose it in `PowerSnapshot.last_error` or an apply-details field.

Suggested tests:

- Fake hardware returning `False` for `write_rapl_msr` and/or `rewrite_epp`; assert the failure is visible in snapshot or result.
- Test that `_last_applied_power_profile` does not mask observed hardware drift after correction failure.

### PP-005: New direct hardware methods bypass the adapter root and are hard to test safely

- Severity: Medium
- Confidence: High
- Category: Testability, safety boundary, platform abstraction
- Introduced by overhaul: Newly introduced
- Affected code:
  - `honor_control/backend/hardware.py:844-896`
  - `honor_control/backend/hardware.py:917-959`
  - `honor_control/backend/hardware.py:988-1025`
  - `honor_control/backend/hardware.py:1027-1034`

Description:

Most `HonorToolsAdapter` read/discovery code uses `self._root` and `_rooted()` so tests can create fake sysfs trees. The new power methods use absolute host paths and real `systemctl` directly. This makes them difficult to unit-test and creates a safety hazard for future tests: a test that constructs a verified fake DMI under `root_path` and calls `stop_competing_power_daemons` would still run host `systemctl stop/mask`.

Evidence and reasoning:

- `stop_competing_power_daemons` imports `subprocess` and calls `systemctl` with no injected runner and writes absolute `/sys/devices/system/cpu/intel_pstate/hwp_dynamic_boost`.
- `rewrite_epp` enumerates absolute `/sys/devices/system/cpu/cpu[0-9]*`, not `self._root / ...`.
- `_write_sysfs` writes `pathlib.Path(path)` directly.
- `write_rapl_msr` opens absolute `/dev/cpu/0/msr`.

Failure scenario:

The current tests do not exercise these methods against fake sysfs/MSR data. A future root-run integration test using `HonorToolsAdapter(root_path=tmp_path)` may unintentionally mask host daemons. More immediately, the MSR encoding bug in PP-002 reached HEAD because there is no pure encoder or injectable file/systemd boundary to test.

Recommended fix:

Separate pure encoding and OS effects:

- Use `self._rooted()` for sysfs/dev paths where possible.
- Inject a command runner for `systemctl`.
- Extract MSR value encoding into a pure function.
- Make daemon stopping a policy decision in `ApplicationService` or a dedicated OS service abstraction, not an unmockable side effect hidden in the hardware adapter.

Suggested tests:

- Unit test `rewrite_epp` against a tmp_path CPU tree.
- Unit test `stop_competing_power_daemons` with an injected runner proving expected commands without invoking systemd.
- Unit test `write_rapl_msr` using an injected MSR read/write object or pure encoder.

## 6. Architecture and maintainability assessment

The overhaul improves several architectural boundaries: frontend code still talks through typed snapshots/intents, config remains centralized, and the hardware adapter remains the only production layer importing `honor.*`. Editable profile definitions are modeled explicitly rather than as loose UI-only state.

The main architectural weakness is that power control now has competing ownership models:

- It wants PPD disabled because PPD overwrites RAPL/EPP.
- It still requires PPD success because `honor.power.apply_profile` and `ppd_ok` remain central to success.
- It publishes applied profile from a service-side memory override, which can diverge from hardware observation after delayed failure or external drift.

This should be resolved with a targeted refactor, not a rewrite. Introduce a small power backend abstraction that separates:

- Profile definition validation
- PPD profile selection, if enabled
- Sysfs governor/EPP application
- RAPL sysfs or MSR application
- Delayed settle/correction and verification
- Observed-state readback

Concrete benefits:

- Tests can validate each mechanism independently.
- PPD can be optional or disabled by policy without making the whole apply partial.
- Delayed correction can report a real lifecycle state instead of being invisible.
- Hardware-specific paths can be injected/rooted, avoiding accidental host writes in tests.

Duplication and coupling:

- `honor-control` now partly duplicates behavior already in `honor-tools/honor/power.py`. That duplication is understandable for the MSR/EPP workaround, but it should be extracted into testable helpers or moved into the dependency with a clear API.
- `HardwarePort` grew low-level methods (`rewrite_epp`, `write_rapl_msr`, `stop_competing_power_daemons`) that are policy-heavy. Stopping/masking daemons is more OS policy than laptop hardware access.

State-management concerns:

- `_last_applied_power_profile` overrides observed profile indefinitely until process restart.
- Delayed rewrite completion does not update state.
- Multiple delayed tasks can coexist but only the last one is tracked.

A targeted refactor is justified around the power apply pipeline and delayed correction lifecycle. A broad rewrite of the whole backend is not justified by the evidence here.

## 7. Positive observations

- The overhaul correctly recognizes that profile application is multi-mechanism and should not be treated as a single opaque command.
- Persisted power profiles include all important knobs: PL1, PL2, governor, EPP, PPD profile, turbo, and max performance.
- Startup reconciliation is a good concept: desired state should be reapplied after restart when hardware drifted.
- The GUI active-profile dirty flag fixes a real usability issue where periodic refresh could snap selection back before the user applies it.
- Platform gating before daemon masking was added, which avoids touching PPD on completely unrecognized machines.
- Existing config validation and immutable snapshot patterns are clean and maintainable.
- The code already treats partial application as a first-class result; the power overhaul should build on that rather than hiding delayed failures.

## 8. Prioritized remediation plan

Must fix before merge or release:

1. Resolve PP-001 by choosing a coherent PPD ownership model. Do not mask PPD while still requiring `powerprofilesctl set` success.
2. Fix PP-002 MSR PL2 encoding and add pure unit tests for the encoder.
3. Fix PP-006 or define the supported Python/runtime matrix so backend/application tests reliably run.
4. Add application tests for startup reconciliation and manual profile apply after initialization.

Should fix soon afterward:

5. Fix PP-003 by cancelling/gating stale delayed rewrites.
6. Fix PP-004 by surfacing delayed rewrite success/failure in result/snapshot state.
7. Fix PP-005 by rooting/injecting sysfs, MSR, and systemd interactions for safe tests.

Optional cleanup and longer-term improvements:

8. Rework documentation compatibility language to match the verified platform gate.
9. Move duplicated low-level Intel power logic into a tested helper module or dependency API.
10. Replace `_last_applied_power_profile or observed` with a state model that distinguishes service-applied, observed, pending correction, and drift.

Validation plan after fixes:

1. Run `ruff check honor_control tests`.
2. Run `python -m pytest tests/test_backend.py::TestCommandQueue -vv`.
3. Run focused new power tests for PPD-disabled, MSR encoding, delayed rewrite ordering, and delayed failure.
4. Run full `python -m pytest` with a CI timeout.
5. On real supported hardware only, perform manual pre-release validation: apply each built-in profile, verify governor/EPP/RAPL readback after settle, reboot and verify startup reconciliation, test AC/battery auto-switch, and verify PPD/systemd state matches the selected ownership model.

## 9. Final verdict

Verdict: **Requires significant fixes**.

The implementation is not safe to release as-is. The minimum changes required to advance to "Ready with minor fixes" are:

- Make PPD lifecycle and profile application internally consistent.
- Correct and test MSR PL2 encoding.
- Restore reliable backend/application tests.
- Ensure delayed rewrite behavior is ordered, cancellable, and observable.

After those are fixed and validated, the remaining issues are mostly maintainability, documentation, and test-depth improvements rather than fundamental blockers.
