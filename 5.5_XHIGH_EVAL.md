# 5.5 XHIGH Power-Profile Overhaul Evaluation

## 1. Executive summary

Final verdict: **Unsafe to merge or release**.

The power-profile overhaul is not release-ready. The most serious issue is architectural: service startup now stops and masks `power-profiles-daemon`, but the profile application path still treats `powerprofilesctl set` success as mandatory. On a supported platform this can make profile application and startup reconciliation fail immediately after the service disables the daemon it still depends on.

The implementation also writes Intel RAPL MSR 0x610 directly, but the PL2 bit construction is wrong: the code sets PL2 enable/time bits as absolute positions and then shifts them by another 32 bits, so the written value does not enable PL2 and loses the preserved PL2 time window. The delayed MSR/EPP rewrite is also fire-and-forget, reports success before these required writes have succeeded, ignores boolean failures, and can leave stale delayed tasks running after later profile changes.

There are good design intentions here: desired/applied/observed state is separated, profile definitions are validated centrally, and the GUI dirty-state fix addresses a real snap-back problem. However, the privileged hardware and systemd behavior needs significant correction and test coverage before this can be considered safe.

## 3. Architecture and behavior overview

The power-profile system has three main layers.

The persisted desired state lives in `ConfigStore` as `PowerState`. It stores the selected manual profile, auto-switch policy, scripts, and a registry of `PowerProfileState` definitions. Defaults come from `POWER_PROFILES` in `honor_control/core/models.py`, with built-in `silent`, `balanced`, and `performance` profiles. Validation of profile names, PL1/PL2 bounds, governor, EPP, PPD profile, turbo, and max-performance percentage is centralized in `config_store._validate_state()`.

The backend orchestrator is `ApplicationService`. `set_power_profile()` validates the requested name, then `_apply_power_profile()` checks `get_power_capability()`, builds a definition dict, and runs `HonorToolsAdapter.apply_power_profile()` through `HardwareCommandQueue`. On success it updates `_last_applied_power_profile`, persists desired state when appropriate, refreshes power state, schedules a delayed `_delayed_power_rewrite()`, and returns success. Auto-switch uses the same `_apply_power_profile()` path with `persist_desired=False`, preserving the manual desired profile while applying AC/battery policy.

The production hardware implementation is `HonorToolsAdapter`. Its base application path still delegates to `honor.power.apply_profile()`, which writes RAPL sysfs limits, calls `powerprofilesctl set`, writes EPP, writes governors, and writes turbo/max-perf controls. The overhaul adds three direct methods:

- `stop_competing_power_daemons()` stops and masks `power-profiles-daemon` and `intel_lpmd`, then tries to enable `hwp_dynamic_boost`.
- `write_rapl_msr()` writes PL1/PL2 directly to `/dev/cpu/0/msr`, MSR 0x610.
- `rewrite_epp()` writes governors and EPP directly under `/sys/devices/system/cpu/cpu*/cpufreq`.

Startup now loads config, refreshes all domains, runs `stop_competing_power_daemons()` through the queue, and reconciles the persisted desired profile if the snapshot says the applied profile differs. Refresh reads live power state through `honor.power.get_status()`, maps the PPD profile back to one of the built-in names, and overlays desired state and in-memory `_last_applied_power_profile`.

The GUI power page receives typed snapshots. The overhaul added `_active_combo_dirty` so a user-selected active profile is not overwritten by the 5-second refresh loop before the user clicks Apply. On a snapshot where the selected combo value matches the applied profile, the dirty flag is cleared.

Error handling is mixed. The initial `honor.power.apply_profile()` result is converted into `governor_ok`, `epp_ok`, `ppd_ok`, `rapl_ok`, and `misc_ok`; any false value produces a partial result. The delayed direct MSR/EPP rewrite is different: it runs after the API has already returned success, ignores the boolean return values from `write_rapl_msr()` and `rewrite_epp()`, and only logs exceptions.

## 4. Findings

### Critical

#### PP-001 - Startup disables PPD before the apply path that still requires PPD

- Severity: Critical
- Confidence: High
- Category: Functional regression, state transition, daemon/service behavior
- Status: Newly introduced by the overhaul
- Affected code:
  - `honor_control/backend/application.py:155-165`
  - `honor_control/backend/hardware.py:844-885`
  - `honor_control/backend/hardware.py:806-840`
  - Dependency context: `../honor-tools/honor/power.py:175-219`

Problem:

`initialize()` now calls `stop_competing_power_daemons()` before `_reconcile_power_profile()`. On a supported Honor platform, that method runs `systemctl stop` and `systemctl mask` for `power-profiles-daemon`. The primary apply path still delegates to `honor.power.apply_profile()`, which calls `powerprofilesctl set` and returns `ppd_ok = False` when that fails. `ApplicationService._apply_power_profile()` treats `ppd_ok` as mandatory for success.

Evidence and reasoning:

- Startup order is refresh, stop/mask daemons, then reconcile.
- `stop_competing_power_daemons()` masks `power-profiles-daemon`.
- `honor.power.apply_profile()` still calls `powerprofilesctl set`.
- `HonorToolsAdapter.apply_power_profile()` converts `raw.get("ppd_ok") is True` into `ppd_ok`.
- `_apply_power_profile()` only succeeds when `governor_ok and epp_ok and ppd_ok and rapl_ok and misc_ok`.

Realistic failure scenario:

On a verified MRA-XXX laptop, the service starts, masks PPD, then tries to reconcile a persisted `performance` profile. `powerprofilesctl set performance` cannot talk to a stopped/masked PPD service, so `ppd_ok` is false. The profile apply returns partial, the delayed direct rewrite is not scheduled, and the user's persisted desired profile is not applied. Manual profile changes after startup have the same failure mode.

Recommended fix:

Choose one ownership model. Either keep PPD running and coordinate with it, or stop depending on `powerprofilesctl` when PPD is intentionally disabled. If disabling PPD remains required, remove `ppd_ok` from the success gate after PPD has been deliberately disabled, make `ppd_profile` optional/diagnostic, and derive applied state from the direct controls. Prefer not masking PPD at startup at all; if needed, make it an explicit admin-controlled setting with rollback.

Suggested tests:

- Unit test `initialize()` on a production-style fake adapter where `stop_competing_power_daemons()` makes later `apply_power_profile()` return `ppd_ok=False`; assert reconciliation does not silently fail.
- Adapter test proving the selected PPD ownership mode: if PPD is disabled, a profile with all direct writes successful should return success without requiring `powerprofilesctl`.
- Integration-style fake test covering startup with persisted `performance` and observed `balanced`.

#### PP-002 - Service startup persistently masks system daemons without user action or rollback

- Severity: Critical
- Confidence: High
- Category: Privilege boundary, OS integration, recovery
- Status: Newly introduced by the overhaul
- Affected code:
  - `honor_control/backend/application.py:155-165`
  - `honor_control/backend/hardware.py:844-897`
  - `packaging/polkit/org.honorlinux.control.policy:24-34`

Problem:

The root service now performs a persistent system configuration change on startup: it masks PPD and intel_lpmd. This is not tied to an explicit user profile change, not gated by the admin-tier `configure-power` authorization, not persisted as a managed desired state, not surfaced in snapshots or operation results, and not reversed on shutdown or uninstall.

Evidence and reasoning:

`systemctl mask` is persistent by design. The code comments explicitly state it is used so daemons stay masked and survive reboots. The polkit policy treats ordinary profile switching as active-local-user level and power configuration as admin level, but this new startup behavior occurs without a D-Bus caller or polkit authorization.

Realistic failure scenario:

A user installs or starts `honor-control` to inspect status. The service detects a supported Honor platform and masks the desktop power manager. GNOME/KDE power settings, other power tooling, and distribution update expectations may break. Stopping `honor-control` does not restore the original system state.

Recommended fix:

Do not mask unrelated system daemons as an implicit startup side effect. If this is truly required for a specific experimental direct-control mode, model it as explicit admin-authorized configuration, record prior enabled/masked state, expose it in diagnostics, and implement rollback/unmask on disable/uninstall. Prefer transient `systemctl stop` only for the profile apply window, or better, integrate through the platform power management service instead of disabling it.

Suggested tests:

- Test that service startup never calls `systemctl mask`.
- If an explicit daemon-management feature is added, test authorization, persisted desired state, rollback, and idempotency for pre-existing masked/disabled daemons.

### High

#### PP-003 - RAPL MSR PL2 enable/time bits are written to the wrong positions

- Severity: High
- Confidence: High
- Category: Functional bug, hardware interface
- Status: Newly introduced by the overhaul
- Affected code:
  - `honor_control/backend/hardware.py:1007-1019`

Problem:

`write_rapl_msr()` builds `hi` for the upper half of MSR 0x610, but sets `1 << 47` and `tw2 << 49` before shifting `hi << 32`. Those are already absolute bit positions for the final 64-bit value. After the shift they land past bit 63 and are masked away. The final value contains the PL2 limit bits but does not set the PL2 enable bit and loses the preserved PL2 time window.

Evidence and reasoning:

The diagnostic arithmetic reproduction with `pl2_units=440` and `tw2=17` produced:

```text
current value:        0x1b800558118
pl2_enable:          False
pl2_time:            0
corrected value:     0x2281b800558118
correct_pl2_enable:  True
correct_pl2_time:    17
```

The lower-half builder uses relative bit positions. The upper-half builder should likewise set PL2 enable as `1 << 15` inside `hi`, and PL2 time as `tw2 << 17`, before shifting the whole `hi` into bits 32-63.

Realistic failure scenario:

A user applies `performance` expecting 55W burst PL2. The direct MSR rewrite writes a PL2 limit but leaves PL2 disabled or time-window zeroed. Burst behavior is not applied as intended, and the direct write can also perturb package limit behavior in ways that are hard to diagnose from the UI.

Recommended fix:

Build PL2 fields relative to the high 32-bit half, or build the final 64-bit value directly with absolute positions once. Preserve unrelated bits that should survive, especially lock/reserved fields where applicable. Consider using a small named-field helper for MSR 0x610 instead of open-coded shifts.

Suggested tests:

- Pure unit test for MSR 0x610 encoding using known `MSR_RAPL_POWER_UNIT`, old package-limit value, expected PL1/PL2 units, enable bits, and time windows.
- Test boundary values and assert no fields spill past bit 63.
- Test that PL1/PL2 fields preserve unrelated old bits intentionally, or explicitly document and test any fields that are reset.

#### PP-004 - Delayed direct writes can fail after the API has already reported success

- Severity: High
- Confidence: High
- Category: Error handling, state synchronization, service sandbox
- Status: Newly introduced by the overhaul
- Affected code:
  - `honor_control/backend/application.py:367-421`
  - `honor_control/backend/hardware.py:898-1045`
  - `packaging/systemd/honor-control.service:19-31`

Problem:

The new delayed rewrite is described as required to make RAPL/EPP values stick, but `_apply_power_profile()` returns success before it runs. `_delayed_power_rewrite()` does not inspect the boolean return values from `write_rapl_msr()` or `rewrite_epp()`. `write_rapl_msr()` returns `False` for missing `/dev/cpu/0/msr`, missing msr module, or permission failure, but the caller logs nothing unless an exception escapes.

Evidence and reasoning:

The service unit has a narrow `CapabilityBoundingSet` and no `CAP_SYS_RAWIO`. MSR access commonly requires that capability even for root. The unit also has hardening that may affect kernel/sysfs writes. Therefore a realistic production failure is that the initial honor-tools sysfs path returns success, but the delayed direct MSR path fails. The public operation result and snapshot can still say the profile was applied.

Realistic failure scenario:

The user applies `performance`. The D-Bus method returns success. Half a second later, `/dev/cpu/0/msr` open/write fails because the msr module is absent or the service lacks required capability. The UI continues to report success, but sustained/burst limits are not actually enforced by the direct write.

Recommended fix:

Do not report final success until all required writes have succeeded, or split the result into "initial apply accepted" and "hardware convergence verified" states. At minimum, inspect the boolean results, update snapshot health/stale diagnostics, and surface delayed failure in `OperationResult.details` or a later event. Align the systemd capability set and sandbox with the hardware operations the service is expected to perform.

Suggested tests:

- Unit test where `write_rapl_msr()` returns `False`; assert the service records a failed/degraded power state.
- Test where `rewrite_epp()` returns `False` for one CPU and verify user-visible diagnostics.
- Packaging test or documented manual test confirming the installed unit can open/write `/dev/cpu/0/msr` on supported hardware.

#### PP-005 - Stale delayed rewrite tasks can race later profile changes and are not all canceled

- Severity: High
- Confidence: Medium
- Category: Concurrency, stale state, lifecycle management
- Status: Newly introduced by the overhaul
- Affected code:
  - `honor_control/backend/application.py:181-186`
  - `honor_control/backend/application.py:367-421`

Problem:

Every successful profile apply overwrites `self._pending_power_task` with a new delayed task. The previous task is not canceled before replacement. Shutdown cancels only the latest task reference. Older tasks can still wake up, enqueue stale RAPL/EPP writes for an earlier profile, and interleave with later profile applies because `_delayed_power_rewrite()` does not hold `_mutation_lock`.

Evidence and reasoning:

The task captures a `PowerProfileState` object, sleeps for 0.5s, then writes that captured profile. There is no generation check, no comparison with current desired/applied profile, and no cancellation of the previous task.

Realistic failure scenario:

A user applies `performance`, then quickly applies `silent`. The first delayed task can still write performance RAPL/EPP after the silent apply has begun or completed. The second delayed task may correct it later, but there is a window of wrong power state. If the later apply fails or the service shuts down, the stale task can be the last write.

Recommended fix:

Cancel and await any previous pending rewrite before scheduling a new one, or use a monotonic generation token and have delayed tasks no-op when they are not the latest generation. Consider running the delayed convergence inside the serialized mutation flow if it is required for correctness.

Suggested tests:

- Fake hardware test applying profile A then B within 0.5s; assert only B's delayed write executes.
- Shutdown test with two pending tasks; assert no stale hardware calls happen after shutdown begins.

#### PP-006 - Power capability does not check the new required resources

- Severity: High
- Confidence: High
- Category: Capability modeling, platform assumptions
- Status: Newly introduced/worsened by the overhaul
- Affected code:
  - `honor_control/backend/hardware.py:597-619`
  - `honor_control/backend/hardware.py:887-1045`
  - `docs/hardware-support.md:23-29`

Problem:

`get_power_capability()` only checks honor-tools importability, verified platform, and `powerprofilesctl`. The overhaul adds required direct writes to `/dev/cpu/0/msr`, `/sys/devices/system/cpu/intel_pstate/hwp_dynamic_boost`, and per-CPU cpufreq EPP/governor files, but the capability does not probe those resources or permissions.

Evidence and reasoning:

The docs say RAPL/governor/EPP sysfs files are required and every mechanism is checked separately. The implementation does not check the new MSR device, msr module availability, CPU 0 online/device presence, EPP path existence, governor writability, HWP dynamic boost writability, or required capabilities.

Realistic failure scenario:

The UI enables the Power page because `powerprofilesctl` exists. Profile apply returns success from the initial path, but the direct RAPL/EPP convergence silently fails because `/dev/cpu/0/msr` is absent or EPP is not exposed. The user sees a supported feature that cannot deliver the advertised behavior.

Recommended fix:

Expand `get_power_capability()` to return detailed resources and separate statuses for PPD, sysfs RAPL, MSR RAPL, EPP/governor, turbo/max-perf, HWP dynamic boost, and service permissions. Gate direct-write mode on all required resources or downgrade to a clearly reported partial/unsupported mode.

Suggested tests:

- Fake filesystem tests for missing `/dev/cpu/0/msr`, missing EPP files, missing intel_pstate files, and permission-denied paths.
- Snapshot tests asserting capability resources and reason codes are user-actionable.

#### PP-007 - Pre-existing command queue failure blocks validation and would break all hardware operations on this runtime

- Severity: High
- Confidence: High
- Category: General codebase concern, async/threading, test infrastructure
- Status: Pre-existing, not introduced by the power-profile overhaul
- Affected code:
  - `honor_control/backend/command_queue.py:68-124`
  - `tests/test_backend.py:69-73`

Problem:

On the local Python 3.14.6 runtime, `HardwareCommandQueue.run("test", lambda: 42)` times out even though the worker thread runs the lambda immediately. This causes `ApplicationService.initialize()` and queue-dependent tests to hang/fail.

Evidence and reasoning:

The smallest pytest test failed:

```text
FAILED tests/test_backend.py::TestCommandQueue::test_run_executes_function
CommandTimeoutError: Hardware command 'test' timed out after 10.0s
```

A direct diagnostic printed that the worker ran at about 0.0005s, but `q.run()` timed out at 3s. This issue predates the overhaul because `command_queue.py` was not changed in `4d8994a..HEAD`.

Realistic failure scenario:

If this runtime behavior occurs in production, every hardware command routed through the queue will time out, including power profile application, startup reconciliation, battery writes, fan operations, and hardware refreshes.

Recommended fix:

Investigate the queue completion mechanism on Python 3.14. Add CI coverage for the supported Python versions and consider an implementation based on `asyncio.to_thread`, `run_in_executor`, or a well-tested worker loop. If Python 3.14 is not supported, set and test an upper Python bound.

Suggested tests:

- Keep `TestCommandQueue.test_run_executes_function` as a required smoke test across all supported Python versions.
- Add a test proving cross-thread completion wakes the event loop promptly, not only when a timeout timer fires.

### Medium

#### PP-008 - `stop_competing_power_daemons()` can exceed the queue timeout during service startup

- Severity: Medium
- Confidence: Medium
- Category: Timeout handling, startup robustness
- Status: Newly introduced by the overhaul
- Affected code:
  - `honor_control/backend/application.py:162-165`
  - `honor_control/backend/hardware.py:874-885`
  - `honor_control/backend/command_queue.py:25-27`

Problem:

`stop_competing_power_daemons()` can run four `systemctl` commands, each with a 5s subprocess timeout. The command queue default timeout is 10s, and `initialize()` does not catch failures from the `stop_daemons` queue call. Worst case is about 20s of subprocess waiting inside a 10s queue timeout.

Evidence and reasoning:

The code loops over two daemons and two actions. A slow or wedged `systemctl`/systemd D-Bus call can exceed the queue deadline and raise `CommandTimeoutError`, preventing service startup and poisoning the queue until the worker finishes.

Realistic failure scenario:

On a booting system where PID 1 is slow to answer, the service times out during startup daemon masking and fails before exporting D-Bus.

Recommended fix:

Do not run daemon-management side effects in the hardware queue startup path. If kept, use a bounded total timeout smaller than the queue timeout, catch and report startup degradation instead of failing the service, and avoid masking on startup as recommended in PP-002.

Suggested tests:

- Fake subprocess test where every systemctl call times out; assert service initialization does not hang or leave the queue poisoned.

#### PP-009 - Applied-profile observation still depends on PPD even when the overhaul disables PPD

- Severity: Medium
- Confidence: High
- Category: State synchronization, observability
- Status: Newly introduced/worsened by the overhaul
- Affected code:
  - `honor_control/backend/hardware.py:778-804`
  - `honor_control/backend/application.py:1102-1146`
  - `honor_control/backend/hardware.py:844-885`

Problem:

`read_power()` derives `applied_profile` only from PPD's current profile. The overhaul disables/masks PPD, so live observation can no longer report applied profile through that mechanism. `_refresh_power()` overlays `_last_applied_power_profile` only in memory; after a restart it is empty.

Evidence and reasoning:

If PPD is stopped, `honor.power.get_status()` will return an empty `ppd_profile` or fail, and the mapping to `silent`/`balanced`/`performance` produces `""`. Startup reads PPD before stopping it, but subsequent refreshes and later restarts cannot reliably infer applied profile unless a profile was applied in the current process.

Realistic failure scenario:

The service starts with desired `balanced`, sees applied `balanced`, skips reconciliation, then masks PPD. Later refreshes show "Not reported" for applied profile even though the service has changed the machine's power management environment.

Recommended fix:

If direct control is the owner, implement direct observation: read RAPL limits, EPP, governors, turbo, and max-perf, then compare against profile definitions. If PPD remains the owner, do not disable it. Avoid using in-memory `_last_applied_power_profile` as the only source of applied state across refreshes/restarts.

Suggested tests:

- Test `read_power()` with empty PPD profile but matching direct hardware values.
- Restart-style test where desired profile should be reconciled or observed without relying on `_last_applied_power_profile`.

#### PP-010 - New direct hardware methods bypass the adapter's injectable root path

- Severity: Medium
- Confidence: High
- Category: Testability, abstraction leakage, platform isolation
- Status: Newly introduced by the overhaul
- Affected code:
  - `honor_control/backend/hardware.py:887-1045`
  - Existing root abstraction: `honor_control/backend/hardware.py:444-458`

Problem:

`HonorToolsAdapter` accepts `root_path` and many existing probes use `_rooted()` or paths discovered under that root. The new methods use absolute host paths and host `systemctl` directly. That prevents safe fake-root tests and makes these methods risky to invoke in diagnostic contexts that expect root isolation.

Evidence and reasoning:

`rewrite_epp()` globs `/sys/devices/system/cpu/cpu[0-9]*`, `write_rapl_msr()` opens `/dev/cpu/0/msr`, and `stop_competing_power_daemons()` calls host `systemctl` and writes host `/sys/devices/system/cpu/intel_pstate/hwp_dynamic_boost`.

Realistic failure scenario:

A future unit test instantiates `HonorToolsAdapter(root_path=tmp_path)` and calls `rewrite_epp()` expecting fake sysfs writes. Instead it attempts to modify the developer's host CPU sysfs. This also explains why the new behavior has no safe automated coverage.

Recommended fix:

Route all filesystem paths through `_rooted()` or explicit injected roots. Abstract daemon control behind an injectable collaborator. Keep production defaults pointed at `/`, but make fake roots safe.

Suggested tests:

#### PP-012 - `write_rapl_msr()` can leak file descriptors on non-`OSError` paths and catches too narrowly

- Severity: Low
- Confidence: Medium
- Category: Resource lifecycle, error handling
- Status: Newly introduced by the overhaul
- Affected code:
  - `honor_control/backend/hardware.py:988-1036`

Problem:

`write_rapl_msr()` manually opens and closes file descriptors three times. If `os.lseek`, `os.read`, `struct.unpack`, or `os.write` fails between open and close, the descriptor is not closed. The function only catches `OSError`; a short read causing `struct.error` would escape.

Evidence and reasoning:

There is no `try/finally` or context manager around the raw file descriptors. This is low probability for normal `/dev/cpu/0/msr`, but the function is privileged hardware code and should be robust to partial failure.

Realistic failure scenario:

An unusual msr device read failure causes an exception after `os.open()` and before `os.close()`. A delayed task logs the exception at the application layer, but the fd remains open until process exit.

Recommended fix:

Use `with open(..., "rb", buffering=0)`/`with open(..., "wb", buffering=0)` or `try/finally` for every raw descriptor. Catch `OSError`, `struct.error`, and short-read conditions explicitly, returning `False` with diagnostics.

Suggested tests:

- Monkeypatch `os.read` to return fewer than 8 bytes and assert no fd leak and a clean `False`.
- Monkeypatch `os.write` to raise after open and assert close is called.

## 6. Architecture and maintainability assessment

The strongest existing architectural choice is the separation between desired, applied, and observed state. The overhaul attempts to preserve this by persisting desired state in `ConfigStore`, reporting observed data from hardware reads, and tracking in-memory applied state. However, disabling PPD undermines the current observed-state design because live applied-profile detection still depends on PPD.

The biggest responsibility problem is ownership of system power management. The code currently uses three owners at once: PPD through `powerprofilesctl`, honor-tools sysfs writes, and new direct MSR/sysfs writes. The overhaul then disables one owner while keeping it mandatory in the apply contract. A targeted refactor is justified: define one explicit power backend mode, such as `PpdBackend` or `DirectIntelBackend`, with separate capability, apply, observe, and cleanup behavior. This does not require a rewrite of the GUI or D-Bus API; it mainly clarifies backend ownership.

Hardware access needs a testable boundary. Existing adapter design supports a root path for fake sysfs, but the new direct methods bypass it. Introduce small injectable collaborators for filesystem paths, MSR access, and daemon control. The concrete benefit is that critical bitfield and daemon behavior can be tested without touching the host.

The delayed rewrite should not be an untracked side effect. If it is required for correctness, it should become part of a convergence state machine with observable status. If it is best-effort, the UI/result should say so. A generation token or single managed task is enough; no broad redesign is needed.

The capability model should become more granular. A single "supported via powerprofilesctl" capability is no longer accurate for a system that requires PPD, sysfs RAPL, direct MSR, EPP, HWP boost, and service capabilities. More detailed capability reporting will improve both safety and user support.

## 7. Positive observations

- The GUI active-profile dirty flag addresses a real user interaction bug: refresh should not override a pending user selection before Apply.
- Profile definitions are strongly typed and validated before persistence.
- Built-in and custom profile data flow through the same `PowerProfileState` model.
- Desired manual profile is preserved during auto-switch applies via `persist_desired=False`.
- The service has a clear high-level architecture: frontends use D-Bus clients, backend owns hardware, snapshots are immutable, and config is service-owned.
- The systemd unit and docs show serious attention to service hardening, even though the new MSR/sysfs operations now need that hardening revisited.
- The bounds check in `write_rapl_msr()` prevents clearly unreasonable PL1/PL2 values from reaching hardware.

## 8. Prioritized remediation plan

Must fix before merge or release:

1. Resolve the PPD ownership contradiction. Do not mask PPD before using `powerprofilesctl`, or remove PPD from the mandatory apply path when direct mode owns power.
2. Remove implicit startup masking of system daemons. Make daemon control explicit, admin-authorized, reversible, and observable if it remains.
3. Fix and test MSR 0x610 encoding, especially PL2 enable/time fields.
4. Make delayed RAPL/EPP convergence observable and failure-aware, or include it in the synchronous apply result.
5. Cancel/gate stale delayed rewrite tasks.
6. Add capability checks for every required direct-write resource and required service permission.

Should fix soon after:

1. Replace PPD-derived applied-profile observation when direct mode is active.
2. Make new hardware methods respect injectable roots or injected path providers.
3. Fix command-queue compatibility on Python 3.14 or constrain supported Python versions.
4. Add active-profile combo dirty-state GUI tests.

Optional cleanup and longer-term improvements:

1. Split power control into explicit backend modes with clear ownership.
2. Add a pure `RaplPowerLimit` encoder/decoder helper.
3. Improve diagnostics so debug bundles include power backend mode, resource checks, and last convergence failure.
4. Reconcile docs with actual direct-control behavior and rollback requirements.

Validation plan after fixes:

1. Run `ruff check honor_control tests`.
2. Run the full pytest suite on every supported Python version.
3. Run targeted fake-root tests for PPD-disabled and PPD-managed modes.
4. Run MSR encoder tests with known vectors.
5. On real supported hardware, perform a manual pre-release gate that verifies PPD state, direct MSR permissions, RAPL readback, EPP readback, reboot/restart reconciliation, and uninstall/rollback behavior.

## 9. Final verdict

Final verdict: **Unsafe to merge or release**.

Minimum changes required to advance to "Requires significant fixes":

- Remove the startup daemon-masking behavior or make it explicit and reversible.
- Fix the PPD ownership contradiction so profile apply can succeed in the chosen mode.
- Correct and test the RAPL MSR bit layout.
- Surface delayed direct-write failures and prevent stale delayed tasks.
- Add tests for the new startup, delayed rewrite, and direct hardware paths.

The overhaul should not be released until those issues are addressed and validated against both fake hardware and a real supported machine.
